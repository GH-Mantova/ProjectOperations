# Source-of-Truth Documents — Registry & Policy

This document defines which files are **source of truth** for ProjectOperations,
the rule that they must be read in full, and the requirement that each one opens
with a thorough, auto-maintained table of contents.

## Registry

These files are the canonical source of truth. Nothing outside this list should
be treated as authoritative; if two sources disagree, the file below wins.

| Document | Location | Owner / update path |
|---|---|---|
| Project instructions | `project_instructions.md` (root) + GitHub blob | Update via `chore/update-instructions-*` PR |
| Roadmap | `roadmap.md` (root) | doc-reconcile PR only |
| Progress log | `progress.md` (root, append-only) | doc-reconcile PR only |
| Claude Code rules | `CLAUDE.md` (root) | Normal PR |
| **Data model relationship map** | `docs/data-model/relationship-map.md` + `.json` | Auto-generated; regenerated by any schema PR |
| **Data model metadata catalog** | `docs/data-model/metadata-catalog.json` | Edited by any schema PR |
| This registry | `docs/SOURCE-OF-TRUTH.md` | Normal PR |

## Rule 1: read in full

When the system — meaning Claude, Claude Code, or any agent/chat — needs to read
a source-of-truth document, it reads the document **in full**, not a fragment.
These documents are the shared memory of the project; partial reads are how
things get missed.

The table-of-contents requirement below exists precisely so that "read in full"
stays affordable in time and tokens: the reader can scan the TOC first, confirm
scope, and jump, rather than re-deriving structure every time.

## Rule 2: every source-of-truth doc opens with a table of contents

Every document in the registry MUST begin (immediately after its H1 title) with
a thorough table of contents covering all of its second- and third-level
sections. The TOC lives between machine markers so it can be regenerated
deterministically:

```
<!-- TOC:START -->
...generated...
<!-- TOC:END -->
```

Generate or refresh a document's TOC with:

```bash
node scripts/data-model/build-toc.mjs <file.md>
node scripts/data-model/build-toc.mjs --check <file.md>   # CI: fail if stale
```

The generator is idempotent and only touches the block between the markers —
running it never disturbs the rest of the document.

## Rule 3: update the TOC on every change

Any PR that changes the heading structure of a source-of-truth document MUST
regenerate that document's TOC in the same PR (`build-toc.mjs <file>`), and the
CI `--check` mode enforces it. The data-model map regenerates its own TOC as part
of `build-relationship-map.mjs`, so schema PRs get this for free.

## Retrofitting the existing large docs (mid-flight safe)

`project_instructions.md`, `roadmap.md`, and `progress.md` predate this policy and
do not yet carry TOC markers. `roadmap.md` and `progress.md` are owned by the
periodic **doc-reconcile PR** and are actively appended to by parallel work, so
they must NOT be rewritten by an unrelated feature branch — that is exactly the
kind of edit that collides with mid-flight PRs.

Retrofit sequencing (safe order):

1. Land this policy + the two generator scripts on their own branch (no edits to
   the big owned docs). Low risk — new files only.
2. Add TOC markers to `CLAUDE.md` and `project_instructions.md` in a small,
   dedicated `chore/toc-*` PR when no PR touching those files is open.
3. Add the TOC to `roadmap.md` and `progress.md` **inside the next scheduled
   doc-reconcile PR only** — never in a feature branch — since that PR already
   owns those files and their `Last updated:` headers.

## Weekly source-of-truth sweep

A scheduled agent runs once a week (00:00 Monday, Australia/Brisbane) and does a
read-only sweep comparing the source-of-truth documents against actual repo
state. It reports drift; it does not auto-edit code or the owned docs. See
`docs/data-model/weekly-sweep.md` for exactly what it checks and how it reports.
