# Smart Wizard — metadata-driven dashboard builder (reads the catalog at RUNTIME)

Marco's decision (BACKLOG-DECISIONS.md #4): the wizard must behave almost like a LIVE agent — it
reads the metadata catalog **at runtime**. **No code generation. No baked-in model list. No
compile-time snapshot.** Add a model to the catalog and the wizard offers it with NO deploy. If you
find yourself generating code from the catalog, you have built the wrong thing.

## What to build (first slice: the runtime-reading shell)

1. **Serve the catalog at runtime.** `docs/data-model/metadata-catalog.json` is generated by
   `scripts/data-model/build-relationship-map.mjs` and is NOT committed. Add a small API seam
   (e.g. `GET /meta/catalog` in a metadata module) that returns the current metadata catalog at
   runtime — generated/read on the server, never bundled into the client at build time.
2. **The wizard** (`apps/web/src/**`, name it `SmartWizard*`): fetches the catalog from that endpoint
   on load and lets the user pick a model → fields → metric/grouping → chart type, producing a
   **dashboard-widget config object** (reuse the existing `customWidget` / dashboard-widget config
   shape). Adding a model to the catalog must make it appear in the wizard with no rebuild.
3. Wire it as an entry in the dashboard builder UI. Keep it to the shell + config output this slice.

## Do NOT
- Do NOT generate TypeScript/config files from the catalog. Runtime read only.
- Do NOT hardcode any model or field list. Everything comes from the catalog at runtime.
- Do NOT block on catalog quality — reviewing the 0/195 metadata coverage is a SEPARATE parallel task;
  the wizard works against whatever the catalog currently holds and improves as it does.
