Remaining High-Value Tendering Directions

Phase 1: Production Hardening From Real Usage

Goal:
Use live pilot feedback to improve the module only where real user friction appears.

Focus:

1. Friction in relationship cards
   - remove any remaining duplicated content
   - tighten cards if users feel they are still too tall
   - improve scanability before adding more information

2. Friction in communication rail
   - ensure the queue is useful and not noisy
   - ensure overdue / upcoming emphasis matches how estimators actually work
   - avoid reintroducing redundant summary cards

3. Friction in stage movement
   - confirm drag/drop and modal prompts still feel safe and fast
   - only add more lifecycle UX if real users hesitate or make mistakes

Why this first:
   - Tendering is already strong enough to pilot
   - the highest-value next improvements should come from runtime evidence, not assumption

Phase 2: Stage Readiness And Governance

Goal:
Make lifecycle progression more operationally meaningful without overcomplicating the experience.

Potential build items:

1. Stronger stage-readiness messaging
   - clearer distinction between required and recommended data
   - more obvious blockers before key transitions

2. Conversion readiness panel refinement
   - keep tender progression separate from job-conversion readiness
   - clearly show what is still missing before conversion

3. Optional `stageEnteredAt` support later
   - only if heuristic stage-age becomes a practical limitation

Why:
   - Pipedrive-style value comes from meaningful movement, not just prettier cards

Phase 3: Richer Activity Model

Goal:
Decide whether the current adapter-backed unified activity approach remains sufficient.

Potential next step:

1. Improve the existing adapter-backed activity experience first
   - richer editing
   - clearer type semantics
   - stronger owner/due controls

2. Only later evaluate a persisted `TenderActivity` model if needed
   - do not introduce schema risk unless the current model becomes a real blocker

Why:
   - the current activity approach has delivered substantial value already
   - persistence changes should be justified by clear operational need

Phase 4: Spreadsheet-Grade Register Operations

Goal:
Make the register more operational for power users.

Potential build items:

1. Configurable visible columns
2. Better row sorting and saved sort behavior
3. Bulk actions
4. Export filtered results
5. Saved filter presets

Why:
   - this is one of the most CRM-like operational advantages still not fully delivered

Phase 5: Contact / Communication Deepening

Goal:
Make Tendering more relationship-aware if pilot users need deeper contact context.

Potential build items:

1. Richer contact panel
   - role in tender
   - phone / email / company
   - relationship history

2. Communication timeline
   - person-level and tender-level activity visibility

3. Clearer commercial role distinctions
   - procurement contact
   - approver
   - reviewer
   - awarded party

Why:
   - only worth doing if current stakeholder cards and communication queue prove too light in real use

Phase 6: Persistent Custom Fields

Goal:
Support business-specific Tendering data without hardcoding every field.

Important caution:
   - this remains one of the riskier future changes
   - only approach it after runtime stability and live adoption are comfortable

Potential build items:

1. Persistent Tender custom fields
2. Admin configuration
3. Visibility in:
   - create
   - workspace
   - list/register columns
   - import mapping
   - filters

Phase 7: Import And Document Workflow Maturity

Goal:
Improve operator confidence in imports and document readiness.

Potential build items:

1. Better import mapping UX
2. Better duplicate handling
3. Stage-aware document readiness or checklists
4. Safer conversion carry-forward controls

Recommended Current Priority Order

1. Production hardening from pilot feedback
2. Stage-readiness refinement if users need stronger lifecycle guidance
3. Register/list operational power
4. Richer communication/contact context only if live use demands it
5. Persistent custom fields later
6. Import/document maturity later

What Not To Do Casually

- do not recombine popup and dedicated workspace shells
- do not rewrite conversion plumbing without runtime evidence
- do not assume a schema migration is the next best step just because activity is important
- do not reintroduce tall stacked admin forms into Tendering surfaces
